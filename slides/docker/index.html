Containers — what & why

A container is a lightweight, isolated process running on the host kernel — packaged with the exact libraries and OS files it needs. They exist because shipping an app means shipping more than code.
env mismatch

Same code, different result

Python version, system libs, and dependency pins differ between laptops, staging, and prod. The app runs here, breaks there.

setup tax

Half a day to get running

A new teammate burns the morning installing Python, system libs, and drivers before running their first pytest.

Containers fix this by packaging code + every dependency + the OS layer into one immutable artifact that runs identically on any machine with a container runtime.

Dockerfile + docker build

A Dockerfile is a text script. docker build compiles it top-to-bottom into an image — with each directive adding one layer. 
DockerfileFROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN  pip install --no-cache-dir \
       -r requirements.txt

COPY . .

EXPOSE 8000

ENTRYPOINT ["uvicorn"]
CMD ["app.main:app",
     "--host", "0.0.0.0",
     "--port", "8000"]
FROMstart from a known base image — here a slim Debian + Python 3.11.
WORKDIRset the working directory inside the image; subsequent commands run here.
COPY requirements.txtcopy only the manifest first — for layer caching (see callout below).
RUNexecute a build-time command — installs dependencies into the image.
COPY . .copy the application source after dependencies are installed.
EXPOSEdocument which port the container will listen on. Metadata only.
ENTRYPOINTthe fixed binary every container starts with. Locks the program — only flags can be overridden.
CMDdefault arguments to the entrypoint. Replaced when caller passes args to docker run.
terminal# build an image from ./Dockerfile, tag it myapp:0.1.
# the trailing "." is the build context — the directory sent to the Docker daemon.
docker build -t myapp:0.1 .
WHY COPY requirements.txt FIRST
Build cache invalidates from the first changed line down. If you COPY . . before installing dependencies, every code change re-runs pip install — slow. Copying the manifest first means dependencies cache until requirements.txt itself changes.

.dockerignore — keep the build context lean

The build context is everything in the folder you pass to docker build. Without filtering, every file in that folder gets uploaded to the Docker daemon — and anything matched by COPY . . ends up baked into the image. .dockerignore is the filter. Same syntax as .gitignore.
Without .dockerignore 
build context (no filter)
# every file in the project folder
app.py
requirements.txt
Dockerfile
.env                    ← contains API keys
.git/                   ← entire history
.venv/                  ← gigabytes of pip cache
__pycache__/            ← stale bytecode
node_modules/           ← if you have any
notes.md, *.log, .DS_Store
Slow upload, bloated image — and a careless COPY . . just shipped your .env to Docker Hub.
With .dockerignore

# Python noise

# dev tooling, caches, docs

README.md

Smaller context, faster build, secrets stay on the host.
WHEN IT RUNS — AND WHEN IT DOESN'T
.dockerignore is read by docker build, not docker push. By the time you push, the image is already built — ignored files were never in it. The same file does nothing for docker run. It's purely a build-time filter on what gets sent to the daemon.

docker run — common flags

docker run turns an image into a running process. Most of its flags answer the same question — how should this isolated process talk to the outside world? Dozens exist; the ones below cover almost every workload.

           -v ./data:/data --name rag-api -d myapp:0.1
FlagWhat it doesTypical example
-p HOST:CONTAINERPublish a container port to a host port.-p 8000:8000
-dDetached — run in background and return.docker run -d assistant
--rmAuto-remove container on exit (no cleanup).--rm assistant

-e KEY=VALSet one environment variable.-e LOG_LEVEL=debug
--env-fileLoad env vars from a file (don't bake secrets).--env-file .env

COMMON COMBOS
Daemon-style, production-ish run
$ docker run -d --name api --restart unless-stopped \
             --env-file .env -p 8000:8000 assistant
Throwaway debug shell
$ docker run --rm -it --entrypoint /bin/bash assistant

CMD vs ENTRYPOINT — the override rules

ENTRYPOINT is the program that runs; CMD is the arguments handed to it. The example below shows the three cases — default, override the arguments, override the program. 
worked example# Dockerfile sets:
ENTRYPOINT ["uvicorn"]
CMD        ["app:app", "--port", "8000"]

# 1) Default — ENTRYPOINT + CMD run together
$ docker run img
   effective → uvicorn app:app --port 8000

# 2) Pass args — they REPLACE the CMD; entrypoint stays
$ docker run img other:app
   effective → uvicorn other:app

# 3) --entrypoint flag REPLACES the entrypoint (CMD still appends!)
$ docker run --entrypoint python img
   effective → python app:app --port 8000

# 3b) To swap BOTH, pass new args after the image name
$ docker run --entrypoint python img -c "print('hello')"
   effective → python -c "print('hello')"
WHEN TO PICK WHICH
CMD only — dev or CI where you want to swap the command on each docker run. ENTRYPOINT + CMD — production binaries with tunable flags (servers, CLI tools like aws / terraform).

ARG vs ENV — build-time vs run-time

ARG exists only while docker build is running — gone the moment the build finishes. ENV is baked into the image and available inside the running container. Same idea as CMD vs ENTRYPOINT: which mechanism you use decides who can change the value and when. 
DockerfileARG PYTHON_VERSION=3.11          # only exists during `docker build`
FROM python:${PYTHON_VERSION}-slim
ENV LOG_LEVEL=info               # baked into the image; alive at runtime
CMD ["sh", "-c", "echo $LOG_LEVEL"]
shell — build & run# 1) Default — uses ARG default at build, ENV default at run
$ docker build -t img .                          # → built on Python 3.11
$ docker run img                                  # → info

# 2) Override ARG — only at build time, with --build-arg
$ docker build --build-arg PYTHON_VERSION=3.12 -t img .   # → built on Python 3.12

# 3) Override ENV — at run time, with -e
$ docker run -e LOG_LEVEL=debug img             # → debug

# 4) Try to override ARG at run time — DOES NOTHING (ARG is gone after build)
$ docker run -e PYTHON_VERSION=3.13 img        # → info  (still the same Python the image was built with)
NEVER PASS SECRETS AS ARG OR ENV
--build-arg DB_PASSWORD=… bakes the value into a layer that's visible with docker history. Same risk for ENV SECRET=… in the Dockerfile. Pass secrets at run time via --env-file or a secrets manager.

Non-root USER — security baseline

Containers run as root by default. If anything inside is compromised, the attacker has root in that container. Fix is two lines: create a user, switch to it.
Default
▶ uvicorn · uid=0 (root)
RCE inside → attacker has root inside the container
USER app →
With USER app
▶ uvicorn · uid=1000 (app)
RCE inside → attacker limited to app's permissions
Dockerfile — production shapeFROM python:3.11-slim
WORKDIR /app
COPY --chown=app:app . .

RUN useradd --create-home --shell /bin/bash app   # 1) create a user
USER app                                          # 2) switch to it

CMD ["python", "-m", "app"]
shell — verify it worked# before the fix
$ docker run img whoami         # → root
$ docker run img id             # → uid=0(root) gid=0(root) groups=0(root)

# after — with USER app
$ docker run img whoami         # → app
$ docker run img id             # → uid=1000(app) gid=1000(app)
Ports < 1024. Non-root users can't bind to port 80 directly. Bind to 8000 inside the container and map -p 80:8000 on the host. ECS, Kubernetes, and most prod orchestrators require non-root.

Multi-stage builds — smaller, safer images

Build in a fat stage with compilers; copy only the artefacts into a thin runtime stage. The toolchain stays behind — the final image carries just what you need to run.
Stage 1 — builder
your source
compiled wheels
apt: gcc · libffi · libpq · libssl
python:3.11-slim
→ discarded
→ COPY --from=builder
Stage 2 — runtime
your source
compiled wheels
python:3.11-slim
→ shipped
single-stage on slim base, build deps kept~400 MB
multi-stage — wheels copied, deps stripped~180 MB
Dockerfile.multistageFROM python:3.11-slim AS builder
RUN apt-get update && apt-get install -y --no-install-recommends \
      gcc libffi-dev libpq-dev libssl-dev \
 && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY requirements.txt .
RUN pip install --user -r requirements.txt

FROM python:3.11-slim AS runtime
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
CMD ["uvicorn", "app:app"]
Why apt-get install in the builder? Slim images strip gcc and C headers. Packages with C extensions (psycopg2, cryptography, sometimes numpy) need them to compile. Install just the build libs you need in stage 1, throw them away by not copying them into stage 2. Sizes above assume the slim base; swapping the builder to the fat python:3.11 bloats stage 1 to ~1 GB before stripping — same final runtime size, easier setup, slower CI.

Tagging + registries

A tag is just a label pointing at an image (identified by its digest). One image can carry many tags. A registry is the server that stores those labelled images so other machines can pull them — Docker Hub, GHCR, ECR all work the same way.
one image (by digest) sha256:abc123…
↑ :latest moves over time · dev only
↑ :v1.2.3 semver release · stable per tag
↑ :sha-abc1234 unique per build · for prod 
shell — build & tag# pass -t multiple times to attach all tags at once
$ SHA=$(git rev-parse --short HEAD)
$ docker build \
    -t ghcr.io/<user>/assistant:latest \
    -t ghcr.io/<user>/assistant:v0.1.0 \
    -t ghcr.io/<user>/assistant:sha-$SHA .

# or add a tag to an existing image after the fact:
$ docker tag assistant:latest ghcr.io/<user>/assistant:v0.1.1
shell — login, push, pull# 1) authenticate (GHCR uses a PAT with `write:packages`; CI uses GITHUB_TOKEN)
$ echo $GH_PAT | docker login ghcr.io -u <user> --password-stdin

# 2) push every tag — same image, different labels, registry shares storage
$ docker push ghcr.io/<user>/assistant:v0.1.0
$ docker push ghcr.io/<user>/assistant:sha-$SHA

# 3) pull from anywhere
$ docker pull ghcr.io/<user>/assistant:sha-$SHA
Production rule. Reference images by sha tag, never :latest. The sha tag is immutable — what you tested is what's running, and rollback is one tag change.

Container networking — three things to know

A container's localhost is its own loopback — not the host's, not another container's. Three rules cover every conversation:
1
Container → container (same network)
<service-name>:<port>
e.g. chromadb:8000
2
Container → host machine
host.docker.internal:<port>
e.g. Ollama at host.docker.internal:11434
3
Container → internet
https://<host>
e.g. api.openai.com — bridge NATs out
Internet — reachable by default via the bridge
Your laptop (host)
Ollama on host :11434

port 8000

to host → host.docker.internal:11434
to host → localhost:11434

Why Docker Compose

A real app rarely runs alone — a RAG service needs an API container and a vector DB container. Wiring them by hand with three docker run commands is brittle. Compose declares the whole stack in one YAML file.
Without Compose 
three commands, ordering matters$ docker network create rag-net

$ docker run -d --name chromadb \
    --network rag-net \
    -v chroma_data:/chroma/.chroma \
    chromadb/chroma

$ docker run -d --name rag-assistant \
    --network rag-net \
    -p 8000:8000 \
    -e CHROMA_HOST=chromadb \
    --env-file .env \
    rag-api:0.1
With Compose 
docker-compose.yml — one fileservices:
  rag-assistant:
    build: .
    ports: ["8000:8000"]
    env_file: .env
    environment:
      CHROMA_HOST: chromadb
    depends_on: [chromadb]

  chromadb:
    image: chromadb/chroma
    volumes: [chroma_data:/chroma/.chroma]

volumes:
  chroma_data:
⊕Auto network — services on a private bridge, addressable by name
⊕Single up / down — docker compose up · docker compose down -v
⊕Start order — depends_on + healthchecks gate dependents
⊕Same file — laptop, CI, single-host prod — identical topology

Compose file walkthrough

Each top-level services: entry becomes a container. Compose also gives every service a private DNS name on a shared internal network — so the api can reach the DB at chromadb:8000 without any port mapping. 
docker-compose.ymlservices:

  rag-assistant:
    build: .                              # build from local Dockerfile
    image: rag-api:0.1
    ports:
      - "8000:8000"                    # host:container — expose to host
    env_file: .env
    environment:
      CHROMA_HOST: chromadb             # DNS name resolves on private net
      CHROMA_PORT: 8000
    depends_on:
      chromadb:
        condition: service_healthy      # wait until chromadb answers
    restart: unless-stopped

  chromadb:
    image: chromadb/chroma:0.5.0          # ships its own HTTP API (uvicorn inside)
    volumes:
      - chroma_data:/chroma/.chroma     # persistent vector index
    healthcheck:                          # /api/v1/heartbeat is built into chroma
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/v1/heartbeat"]
      interval: 5s
      retries: 5
    restart: unless-stopped

volumes:
  chroma_data:
services:each child is one container on the same private network.
build: .Compose runs docker build using the Dockerfile in the current directory.
image:tag the build (or pull a pre-built image when there's no build:).
ports:host:container — only services that talk to the outside need this.
env_file:load every variable from .env at startup.
depends_on + condition:service_healthy blocks the api until chromadb's healthcheck reports OK.
healthcheck:orchestrator probe — distinguishes container running from service responding. Same signal ECS, ALB, k8s use.
volumes:named volume (top-level volumes:) survives down; only down -v wipes it.
docker compose up --buildbuild images and start everything
docker compose logs -f rag-assistantfollow logs from one service
docker compose down -vstop and remove (also drop volumes)

Recap & resources

You now have a complete mental model: Dockerfile → image → container → Compose. The app, its dependencies, and its companion services travel as one unit.

Key takeaways

Containers ≠ VMs

Namespaced processes on the host kernel. Megabytes of overhead, sub-second startup, same code on any host.

Image is the recipe

Built once from a Dockerfile, stored as immutable layers. Containers are running instances of that frozen image.

Order = cache

Copy requirements.txt before source so the heavy pip install layer caches until dependencies change.

Flags = isolation seams

-p opens a port. -v shares a folder. -e injects config. -d detaches. That's the playbook.

Compose for topology

One YAML file declares every service, volume, and network. Same file on laptop, CI, and single-host prod.

Service DNS is free

Compose puts every service on a private network and resolves them by name.
