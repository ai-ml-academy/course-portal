String Mastery

You already know strings hold text. But in real projects — cleaning data, parsing file paths, formatting reports — you need to slice, split, reshape, and format strings fluently. These four methods handle 90% of all text work.

.split(sep)
Chop into a list of parts
.join(list)
Glue a list back into a string
.strip()
Remove leading/trailing whitespace
.replace(old, new)
Swap all occurrences
.lower() / .upper()
Change case
.startswith()
Check how a string begins
# Real-world example: clean a messy CSV row
raw = " Alice , 92 , A , pass "
fields = [f.strip() for f in raw.split(",")]
# → ["Alice", "92", "A", "pass"]

# Clean a column name for Python
col = "Annual Income ($)"
clean = col.strip().lower().replace(" ", "_")
# → "annual_income_($)"

Strings are immutable — every method returns a new string. The original never changes. That's why you can safely chain: .strip().lower().replace()

String Formatting — f-strings

f-strings go far beyond basic f"Hello {name}". You can control precision, alignment, and padding — essential for clean output in reports, logs, and tables.

SyntaxWhat it doesExampleOutput
{x:.2f}2 decimal placesf"{3.14159:.2f}"3.14
{x:.1%}As percentagef"{0.915:.1%}"91.5%
{x:>10}Right-align, width 10f"{'hi':>10}"        hi

# Formatted table output
models = [("rf", 0.91), ("xgb", 0.94), ("lr", 0.85)]

for name, score in models:
    print(f" {name:<10s} {score:.1%}")

# rf 91.0%
# xgb 94.0%
# lr 85.0%

"f-strings can also contain expressions: f\"Sum: {sum(nums)}\" — any valid Python expression works inside the braces."

Regular Expressions

String methods like .split() and .replace() work for simple text. But when you need to find patterns — "any email address", "all 4-digit numbers", "text between quotes" — you need regex. Think of it as a search language for text.

Python's re module gives you pattern matching superpowers. Here are the building blocks:

\d
Any digit (0–9)
"item42" → "4", "2"
\w
Any word character (letter, digit, _)
"hi_2" → "h","i","_","2"
\s
Any whitespace (space, tab, newline)
"a b" → " "
.
Any character (except newline)
"abc" → "a","b","c"
+
One or more of the previous
\d+ matches "42", "7"
*
Zero or more of the previous
\d* matches "", "42"
?
Zero or one (optional)
colou?r → "color", "colour"
[abc]
Any one of these characters
[aeiou] matches vowels
^...$
Start and end of string
^\d+$ = "only digits"

"Regex is write-once, read-never — always add a comment explaining what your pattern does."

Regex — The re Module

You know the pattern syntax — now here's how to use it in Python. The re module gives you four main tools, each for a different job.

re.search()
Find first match anywhere
re.match()
Match at the start only
re.findall()
Find all matches → list
re.sub()
Find & replace with pattern
import re

text = "Order #1234 shipped on 2024-03-15"

# Find all numbers
re.findall(r"\d+", text)
# → ["1234", "2024", "03", "15"]

# Extract the date
m = re.search(r"\d{4}-\d{2}-\d{2}", text)
m.group()  # → "2024-03-15"

# Replace digits with X
re.sub(r"\d", "X", text)
# → "Order #XXXX shipped on XXXX-XX-XX"
  • • Always use raw strings for patterns: r"\d+" — prevents Python from interpreting backslashes
  • • re.search() returns a Match object (or None). Call .group() to get the text
  • • Use parentheses for capture groups: r"(\d{4})-(\d{2})-(\d{2})" extracts year, month, day separately

Error Handling

In real code, things go wrong — files don't exist, APIs time out, users enter garbage. Instead of your program crashing, try/except lets you catch errors and decide what to do.

try: run risky code
↓
✅ No error
↓
else: success
❌ Error raised
↓
except: handle it
↓
finally: always runs (cleanup)
try:
    value = int("not_a_number")
except ValueError as e:
    print(f"Bad input: {e}")
else:
    print(f"Got: {value}")
finally:
    print("Done.")

else runs only if no error occurred. finally runs no matter what — use it for cleanup (closing files, connections).

Error Handling — Raising & Custom Exceptions

You can also raise your own errors when something is wrong. And for bigger projects, you can define custom exception classes so callers know exactly what failed.

Raising errors

def set_age(age):
    if age < 0:
        raise ValueError(
            "Age can't be negative"
        )
    return age

Custom exceptions

# 1. Define it
class InvalidScoreError(Exception):
    pass

# 2. Raise it
def set_score(score):
    if score > 100:
        raise InvalidScoreError(
            f"Score {score} > 100")
    return score

# 3. Catch it
try:
    set_score(150)
except InvalidScoreError as e:
    print(f"Caught: {e}")
ExceptionWhen it happens
ValueErrorRight type, wrong value ("abc" as int)
TypeErrorWrong type entirely (adding str + int)
KeyErrorDict key doesn't exist
IndexErrorList index out of range
FileNotFoundErrorFile path doesn't exist

"Catch specific exceptions, not bare except: — otherwise you'll silently swallow bugs."

Logging

When debugging, print() is tempting. But in any real project it becomes a mess — you can't filter messages, control output, or turn them off in production. Python's built-in logging module solves all of this.

❌ print() problems

print("starting...")
print("value:", x)
print("ERROR! bad input")
print("done")
  • • No severity levels
  • • No timestamps
  • • Can't turn off without deleting
  • • All goes to stdout

✅ logging solution

import logging

logging.info("Starting...")
logging.debug(f"value: {x}")
logging.error("Bad input")
logging.info("Done")
  • • Each message has a level
  • • Auto timestamps & context
  • • Filter by severity
  • • Route to files, services, etc.

Log Levels (low → high severity)

DEBUG
Detailed diagnostic
INFO
Things working fine
WARNING
Something unexpected
ERROR
Something failed
CRITICAL
App can't continue
# Log to console
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s"
)
# → 2024-03-15 10:30:00 [INFO] Starting...

# Log to a file (add filename parameter)
logging.basicConfig(
    filename="app.log",
    level=logging.DEBUG,
    format="%(asctime)s [%(levelname)s] %(message)s"
)
# All messages now saved to app.log

Rule of thumb: use print() for user-facing output, logging for everything else. Add filename= to save logs to a file instead of the console.

Mutability & The Aliasing Trap

This is the #1 source of silent bugs in Python. When you write b = a for a list or dict, you're not making a copy — you're creating a second name for the same object. Change one, and the other changes too.

❌ The trap

a = [1, 2, 3]
b = a       # alias, NOT a copy!

b.append(99)
print(a)   # → [1, 2, 3, 99] 😱
a
→
[1, 2, 3, 99]
b
↗

Both point to the SAME object

✅ The fix

a = [1, 2, 3]
b = a.copy() # new object!

b.append(99)
print(a)   # → [1, 2, 3] ✅
a
→
[1, 2, 3]
b
→
[1, 2, 3, 99]

Each has its OWN object

"= shares. .copy() duplicates. Know which one you're doing."

Shallow vs Deep Copy

.copy() works for flat data. But when you have nested structures — lists inside dicts, dicts inside dicts — a shallow copy only copies the outer container. The inner objects are still shared. Use deepcopy() for true independence.

Shallow copy — inner list shared

config = {"layers": [64, 32]}
exp = config.copy()

exp["layers"].append(16)
print(config["layers"])
# → [64, 32, 16] ← corrupted!
→

Deep copy — fully independent

from copy import deepcopy

config = {"layers": [64, 32]}
exp = deepcopy(config)

exp["layers"].append(16)
print(config["layers"])
# → [64, 32] ✅ safe
OperationWhat it doesUse when
b = aAlias (same object)You want both names to share
b = a.copy()Shallow (new container, shared contents)Flat data (no nesting)
b = deepcopy(a)Deep (everything copied)Nested data (lists in dicts, etc.)

⚠️ Also beware the mutable default argument trap: def f(x=[]) shares the same list across all calls. Use def f(x=None) instead.

Collections Module

Python's built-in dict and list handle most things. But certain patterns — grouping, counting, sliding windows — come up so often that the collections module provides specialized tools that eliminate boilerplate and prevent bugs.

defaultdict — auto-create missing keys

from collections import defaultdict

groups = defaultdict(list)
for name, dept in employees:
    groups[dept].append(name)
# No KeyError, no if-checks

Like a filing cabinet that auto-creates a new folder the first time you use it.

Counter — frequency counting in one line

from collections import Counter

labels = ["cat", "dog", "cat", "bird"]
counts = Counter(labels)
counts.most_common(2)
# → [("cat", 2), ("dog", 1)]

Returns 0 for missing items (no KeyError), supports arithmetic between Counters.

"defaultdict for grouping. Counter for counting. Reach for these before writing manual loops."

Iterators & Generators

A list stores everything in memory at once. A generator produces items one at a time, on demand — like a conveyor belt that only makes the next item when you ask for it. This is called lazy evaluation, and it's how you process data that's too large to fit in memory.

The yield keyword pauses the function and hands back a value. Next call, it resumes where it left off.

function →
1
→
2
→
3
→
?
→
?
…produced on demand

List: everything upfront

def get_squares(n):
    result = []
    for i in range(n):
        result.append(i ** 2)
    return result
# Stores ALL n items in memory

Generator: one at a time

def gen_squares(n):
    for i in range(n):
        yield i ** 2

# Only 1 item in memory at a time
# Use like: for x in gen_squares(n)
  • • Generator expression: (x**2 for x in range(n)) — like list comp but with ()
  • • ⚠️ Generators are one-shot — once exhausted, they're empty. Loop again = nothing happens
  • • itertools module has power tools: chain(), islice(), compress() for composing lazy pipelines

Generators — itertools

The itertools module gives you building blocks for chaining generators into lazy data pipelines — process millions of rows without loading everything into memory.

chain(a, b)
Glue iterables end-to-end
islice(it, n)
Take first n items
compress(data, mask)
Filter by boolean mask
groupby(it, key)
Group consecutive items
from itertools import chain, islice

# chain: glue two lists end-to-end without copying
a = [1, 2, 3]
b = [4, 5, 6]
combined = chain(a, b)  # no new list created!

# islice: take first n items from any iterable
first_4 = list(islice(combined, 4))
# → [1, 2, 3, 4] (only produced 4 items, stopped)

Lazy pipeline: nothing runs until you iterate

# Each step is a generator — no work happens yet
nums = range(1_000_000)             # lazy
squared = (x ** 2 for x in nums)    # lazy
evens = (x for x in squared if x % 2 == 0) # lazy

# Only NOW does work happen — and only for 5 items
result = list(islice(evens, 5))
# → [0, 4, 16, 36, 64]
# The other 999,995 items? Never computed.

"Each stage only produces an item when the next stage asks for it. 1 million items defined, but only 5 ever computed."

Closures

Normally, when a function finishes, its local variables disappear. But what if a function returns another function? The inner function can remember variables from the outer function — even after the outer one is done. That's a closure: a function with a backpack of remembered data.

Step by step:

def make_greeter(greeting): 1 Outer function takes a value
def greet(name): 2 Inner function is defined inside
return f"{greeting}, {name}!" 3 Uses greeting from outer scope
return greet 4 Returns the inner function
# make_greeter is done, but greeting lives on
say_hello = make_greeter("Hello")
say_hey = make_greeter("Hey")

say_hello("Ali")   # → "Hello, Ali!"
say_hey("Sara")   # → "Hey, Sara!"
say_hello(name)
🎒 greeting = "Hello"
say_hey(name)
🎒 greeting = "Hey"
  • • The inner function "closes over" the variable greeting — that's why it's called a closure
  • • Each call to make_greeter creates a brand new function with its own captured value
  • • The outer function is done, but the data lives on inside the returned function's backpack
  • • This is the foundation of decorators — which we'll see next

Decorators

A decorator is a function that wraps another function to add behavior — without changing the original function's code. Think of it as gift wrapping: the gift (your function) stays the same, but the wrapper adds something on top (logging, timing, validation, etc.).

@timer (decorator)
my_function()
adds timing logic around it
import time, functools

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.time()-start:.2f}s")
        return result
    return wrapper

@timer
def train_model(data):
    ...  # @timer wraps this automatically
  • • @timer above a function is shorthand for train_model = timer(train_model)
  • • Always use @functools.wraps(func) inside your decorator to preserve the original function's name and docstring
  • • Common built-in decorators: @property, @staticmethod, @classmethod, @dataclass

Context Managers

You've used with open() for files. The with statement works because files implement a context manager protocol — a setup/cleanup pattern that guarantees resources are released, even if an error occurs. You can write your own for any resource that needs cleanup.

__enter__() — setup (acquire resource)
↓
Your code runs inside the with block
↓
__exit__() — cleanup (always runs, even on error)

Class-based

class Timer:
    def __enter__(self):
        self.start = time.time()
        return self
    def __exit__(self, *args):
        print(f"{time.time()-self.start:.2f}s")
# Using it:
with Timer():
    train_model(data)
# → 2.34s

With contextlib (simpler)

from contextlib import contextmanager

@contextmanager
def timer():
    start = time.time()
    yield  # your code runs here
    print(f"{time.time()-start:.2f}s")
# Using it:
with timer():
    train_model(data)
# → 2.34s

Both do the same thing — class-based is more explicit, contextlib is less code. Files, DB connections, locks, temp directories — anything that needs guaranteed cleanup is a candidate.

Dataclasses

Writing __init__, __repr__, and __eq__ by hand for every class is tedious. The @dataclass decorator auto-generates all of that from your field declarations. It's modern Python's answer to boilerplate classes — use it for configs, records, and DTOs.

❌ Without dataclass (verbose)

class Model:
    def __init__(self, name, acc, lr):
        self.name = name
        self.acc = acc
        self.lr = lr
    def __repr__(self):
        return f"Model({self.name})"
    def __eq__(self, other):
        ...
→

✅ With dataclass (clean)

from dataclasses import dataclass

@dataclass
class Model:
    name: str
    acc: float
    lr: float = 0.01

# __init__, __repr__, __eq__
# all auto-generated!
# Useful features
@dataclass(frozen=True)       # immutable (can't change fields)
@dataclass(order=True)        # enables sorting with < > etc.

# For mutable defaults, use field():
from dataclasses import field
tags: list = field(default_factory=list) # safe!
  • • frozen=True makes instances immutable — good for configs that shouldn't change after creation
  • • __post_init__() lets you run custom logic after auto-generated __init__ (validation, computed fields)
  • • ⚠️ Never use tags: list = [] as a default — all instances would share the same list. Use field(default_factory=list)
