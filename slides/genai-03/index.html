APIs come in different styles

"API" is a generic term — it just means "a way for two programs to talk". Several styles exist for talking over a network. Each has a different shape, transport, and best fit.

REST

HTTP + JSON

Resources accessed via URLs + a small set of HTTP verbs. Stateless, debuggable with curl.

GraphQL

HTTP + JSON

One endpoint. The client sends a query describing exactly the fields it wants back.

gRPC

HTTP/2 + protobuf

Protobuf contracts, code-generated clients, binary on the wire. Common between microservices.

WebSocket / SSE

long-lived TCP

Persistent connection — server can push events to the client. Used for streaming and live updates.

Transport — how do I get a message from A to B?
Payload format — once it arrives, how is the data shaped so both sides can parse it?
Style Transport Payload format
REST HTTP JSON
GraphQL HTTP JSON — with a query language inside the body
gRPC HTTP/2 Protobuf — binary, schema-defined

REST + HTTP — how they stack

Two separate ideas that combine in every web API you'll touch. Each layer does one thing and hands off to the next.
REST The design style. What a URL means, what a verb does. style
↓expressed via
HTTP The wire protocol. Message format, methods, status codes, headers, body. protocol
↓runs on
TCP / IP The network plumbing. Packets get from your laptop to a server. network

Anatomy of an HTTP message

Every HTTP exchange is the same shape — a start line, some headers, and an optional body. The request asks; the response answers.

HTTP Methods + REST URL — verbs and nouns

In REST the URL is a noun (what) and the method is a verb (how). Five verbs cover almost everything — each with a meaning, an example URL, and an idempotency guarantee.
IDEMPOTENT = SAFE TO RETRY
Calling it twice has the same effect as once. GET and DELETE are safe; POST isn’t — a retried POST /orders creates two orders.

HTTP status codes

The first digit groups outcomes into three families. Always check the status before reading the body — a 500 response usually has no JSON, just an error string.

Without FastAPI manual

from flask import Flask, request, jsonify
app = Flask(__name__)

@app.route("/ask", methods=["POST"])
def ask():
    data = request.get_json()
    if "question" not in data:
        return jsonify(
            {"error": "missing question"}
        ), 400
    if not isinstance(data["question"], str):
        return jsonify(
            {"error": "must be str"}
        ), 400
    top_k = data.get("top_k", 5)
    result = rag.answer(
        data["question"], top_k=top_k
    )
    return jsonify(result.__dict__)

Manual JSON parsing, manual validation, manual error responses, no docs page.

With FastAPI type hints

from fastapi import FastAPI
from pydantic import BaseModel

class AskRequest(BaseModel):
    question: str
    top_k: int = 5

class AskResponse(BaseModel):
    answer: str
    sources: list[str]

app = FastAPI()

@app.post("/ask", response_model=AskResponse)
def ask(req: AskRequest):
    return rag.answer(
        req.question,
        top_k=req.top_k,
    )

Pydantic validates and parses. Errors return a clean 422. /docs is generated automatically.

A FastAPI route — same logic, new transport

Same LLMClient.chat(...), same message list, same multi-turn loop. Only the transport changes — stdin becomes an HTTP route, print() becomes a JSON response, one in-memory list becomes a dict keyed by session_id.

API

app/main.py

from fastapi import FastAPI
from pydantic import BaseModel
from app.llm.client import LLMClient

app      = FastAPI()
client   = LLMClient()
sessions: dict[str, list] = {}

class ChatRequest(BaseModel):
    session_id: str
    message:    str

@app.post("/chat")
def chat(req: ChatRequest):
    history = sessions.setdefault(req.session_id, [SYSTEM_MSG])
    history.append({"role": "user", "content": req.message})
    reply = client.chat(history)
    history.append({"role": "assistant", "content": reply})
    return {"reply": reply}

Automatic API documentation

Your type hints + Pydantic schemas produce a live OpenAPI spec at /openapi.json, rendered as a clickable Swagger UI at /docs.

What these terms mean

OpenAPI

A standardised spec

A JSON file describing every endpoint, parameter, request body, response shape, and error. Industry standard for REST APIs.

served at /openapi.json

Swagger UI

An interactive page

A web interface that reads an OpenAPI spec and renders a clickable list of endpoints — with a "Try it out" button that calls your live server.

served at /docs

ReDoc

An alternative reader

A second viewer for the same spec — cleaner reference layout, no "try it out". Good for sharing with external consumers.

served at /redoc

Multi-turn state — where does conversation history live?

For an LLM to "remember" what was said earlier, every /chat call must include the full conversation history. Where you keep that history decides how the service scales and what survives a restart.

The simplest option — in-process dict

sessions: dict[str, list[dict]] = {}

@app.post("/chat")
def chat(req: ChatRequest):
    history = sessions.setdefault(req.session_id, [SYSTEM_MSG])
    history.append({"role": "user", "content": req.message})
    reply = client.chat(history)
    history.append({"role": "assistant", "content": reply})
    return {"reply": reply, "turns": len(history) // 2}

Fine for a demo — one process, fast lookups, no extra infrastructure. Two limits show up the moment you go past one instance:

Limitations

  1. History gets erased when the process restarts.
  2. Doesn’t scale across instances — each replica has its own dict.

Testing the API — four levels of confidence

There's no single "right" way to test a FastAPI app — you reach for different tools depending on what you're checking. Start at the top, drop down as the question gets sharper.
1 · curl

Sanity check from a shell

curl -s localhost:8000/health

curl -s -X POST localhost:8000/chat \
  -H 'content-type: application/json' \
  -d '{"session_id":"a","message":"hi"}'

No code to write. Fast feedback that the service is up and accepts your payload.

2 · /docs

Interactive try-it-out

FastAPI ships Swagger UI at /docs. Click an endpoint, click Try it out, paste a JSON body, hit Execute. Best for showing colleagues or QA people who don't write code.

Pydantic validation errors show up as 422 with the exact field that failed — no need to dig through tracebacks.

3 · requests

From a Python notebook or script

import requests
r = requests.post(
    "http://localhost:8000/chat",
    json={"session_id": "a", "message": "hi"},
)
print(r.status_code, r.json())

Same call shape as production clients. Drives the notebook walkthrough.

4 · pytest + TestClient

Locked-in regression coverage

from fastapi.testclient import TestClient
import assistant_api
client = TestClient(assistant_api.app)

def test_health():
    assert client.get("/health").status_code == 200

TestClient calls routes in-process — no server needed, no network. CI runs these on every push.
