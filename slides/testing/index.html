What You'll Learn Today

Testing is a skill every software engineer needs. Whatever framework you use, five ideas show up in every test suite — learn them once and every tutorial clicks.

Why

Tests are a safety net

They catch regressions when you change code and let you refactor without fear of silently breaking something.

What

Test behaviour, not numbers

Good tests verify intent and invariants. Bad tests pin down exact values that change with every run.

Where

Tests live in their own directory

A tests/ folder that mirrors your source tree. Naming conventions let the framework find them.

How to run

One command, automatic discovery

You don't register tests. The framework walks the tree, runs everything, and reports pass or fail clearly.

How to scale

Share setup, cover many inputs

Fixtures eliminate duplicated setup. Parametrisation runs one body against many inputs.

How to measure

Know what isn't tested

Coverage shows which lines your suite never executes — so you can decide whether to cover them.

Testing frameworks in Python

Python has several testing frameworks. The two you'll see most often are pytest and unittest. They do the same job with different styles.

stdlib

unittest — ships with Python

No install needed. Part of the Python standard library — available wherever Python is. Test style: classes inheriting TestCase, methods like assertEqual, assertRaises.

community

pytest — install with pip

$ pip install pytest

Third-party package. Test style: plain functions + plain assert. Richer failure messages, huge plugin ecosystem. De-facto standard in modern Python.

🎯 We use pytest in this class

Less ceremony, better failure output, and pytest can run unittest tests too — so you lose nothing by picking it. (We already added it to [project.optional-dependencies].dev in Class 07.)

Where tests live in your project

Tests go in a top-level tests/ directory that mirrors your src/ layout. One test file per source file. A reader can immediately find the tests for any module.

project tree
my_project/
├── pyproject.toml
├── src/
│   ├── data_prep/
│   │   └── preprocessing.py
│   ├── model/
│   │   └── architecture.py
│   └── pipeline.py
└── tests/
    ├── conftest.py               # shared fixtures
    ├── test_preprocessing.py     # tests for data_prep/preprocessing.py
    ├── test_architecture.py      # tests for model/architecture.py
    └── test_pipeline.py          # tests for pipeline.py
💡 Naming convention

Test files start with test_. The rest mirrors the source file name — preprocessing.py becomes test_preprocessing.py. That convention isn't cosmetic: pytest uses it to auto-discover your tests.

What to test, what NOT to test

The most common mistake is pinning down exact numbers. Good tests verify the behaviour of your code — not the specific values it happens to produce on your machine today.

✓ DO test

  • Structural properties — shape, type, dtype, value range.
  • Behavioural contracts — for this input, get that kind of output.
  • Edge cases — empty, single element, zero, negative, None.
  • Error paths — invalid input raises the right exception.

✗ DON'T test

  • Exact numeric values — anything that drifts with seed, hardware, or library version.
  • Implementation details — how the function computes; algorithms can change.
  • External systems — network, APIs, filesystem. Mock them or skip.
  • Trivial code — one-line getters with no logic to catch bugs.
🎯 The principle

Test the intent, not the implementation. If two correct implementations could pass the same test, your test is good. If changing the algorithm breaks the test even though the output is still correct, your test is too tight.

Discovery — pytest finds your tests for you

You don't register tests anywhere. Name files and functions following the convention, put them under a tests/ directory, and pytest walks the tree to collect everything that matches.

The three naming rules

  • Files — test_*.py or *_test.py
  • Functions — test_*
  • Classes — Test* (with methods test_*)
10 / 16

Your first test

  • A pytest test is a function named test_*, using plain assert.
  • Common assertion helpers for things assert can't do cleanly on its own:
    • with pytest.raises(Err): — expect a specific exception
    • == pytest.approx(val) — compare floats with tolerance
    • with pytest.warns(Warn): — expect a warning
  • Related tests can be grouped in a Test* class — pure organisation, no inheritance.
    • src/calculator.py
    def add(a, b):
    return a + b


def divide(a, b):
    if b == 0:
        raise ValueError("divide by zero")
    return a / b
    tests/test_calculator.py
    import pytest
from calculator import add, divide


def test_add():
    assert add(2, 3) == 5


def test_divide_by_zero_raises():
    with pytest.raises(ValueError):
        divide(10, 0)

    Diagnostics — rich failure output

    When an assert fails, pytest introspects the expression and shows the line, both values, and how each side was computed. No print() debugging.

    buggy code + test
    # src/pricing.py — bug: + should be −
def off(price, rate):
    return price + rate * price


# tests/test_pricing.py
def test_20_off():
    assert off(100, 0.2) == 80
    pytest output
    FAILED test_pricing.py::test_20_off

>   assert off(100, 0.2) == 80
E   assert 120.0 == 80
E    +  where 120.0 = off(100, 0.2)

    Three things pytest hands you for free: the expression, both values (120.0 vs 80), and how the value was computed — from a plain assert.

    Fixtures and conftest.py — reusable setup

    Tests need setup: a sample dataset, a configured object, a temp file. A fixture builds it once. Any test can ask for that fixture by listing it as a parameter, and pytest wires it in automatically.

    src/users.py — the code under test
    def oldest(users):
    """Return the user with the highest age."""
    return max(users, key=lambda u: u["age"])

    ① Define the fixture

    tests/conftest.py
    import pytest


@pytest.fixture
def sample_users():
    return [
        {"name": "Alice", "age": 30},
        {"name": "Bob",   "age": 25},
        {"name": "Carol", "age": 35},
    ]

    ② Ask for it in a test

    tests/test_users.py
    from users import oldest


def test_oldest_user(sample_users):
    # pytest passes sample_users in
    result = oldest(sample_users)
    assert result["name"] == "Carol"
    🔑 Why exactly conftest.py?

    The filename is a pytest convention — it must be named conftest.py exactly. pytest walks the directory tree and auto-loads any conftest.py it finds. Fixtures defined there become available to every test in that directory (and its subdirectories) without an import.

    Parametrize — one body, many inputs

    When the same property should hold for several inputs, don't write near-identical tests. @pytest.mark.parametrize runs one body once per input and reports each case separately — same add function we wrote on slide 10, now tested against three input pairs.

    tests/test_calculator.py
    import pytest
from calculator import add


@pytest.mark.parametrize("a, b, expected", [
    (2, 3, 5),
    (0, 0, 0),
    (-1, 4, 3),
])
def test_add(a, b, expected):
    assert add(a, b) == expected
    pytest -v output
    test_add[2-3-5]   PASSED
test_add[0-0-0]   PASSED
test_add[-1-4-3]  PASSED

3 passed in 0.02s

    One function, three independent runs. pytest labels each case with its inputs — if [-1-4-3] ever fails you see exactly which input broke the property.

    💡 Fixtures and parametrize compose

    Use fixtures for expensive setup (a database, a trained model) and parametrize for the varying inputs. A test function can take both.

    14 / 16

    Coverage — what your tests don't touch

    Writing tests is good; knowing what you haven't tested is better. The pytest-cov plugin records which lines of your source code actually executed during the suite. Anything that didn't run is a blind spot.

    shell
    $ pip install pytest-cov
$ pytest --cov=src --cov-report=term-missing
    ⚠️ Coverage is a diagnostic, not a target

    100% coverage with weak assertions catches nothing. 70% coverage with sharp assertions on the critical paths catches real bugs. Use the Missing column to find uncovered error paths worth testing — don't chase the number.

    Plugins & the CLI — extending the suite

    pytest has a huge plugin ecosystem; install any of them with pip install <plugin> and they're picked up automatically. The CLI has a handful of flags you'll use every single day.

    Plugins worth knowing

    pytest-cov

    Coverage measurement.

    pytest-xdist

    Run tests in parallel across cores.

    pytest-mock

    Cleaner syntax for unittest.mock.

    pytest-asyncio

    Support for async def tests.

    pytest-benchmark

    Track performance regressions.

    pytest-randomly

    Shuffle test order each run.

    Daily CLI flags

    pytestrun everything in tests/
    pytest -vverbose — one line per test
    pytest -xstop on first failure
    pytest --lfre-run only last-failing
    pytest -k "name"filter by substring
    pytest file::testrun a single test
    pytest -sshow print() output
    pytest --cov=srcmeasure coverage
