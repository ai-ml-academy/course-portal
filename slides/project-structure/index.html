Project structure

What a well-organised Python project looks like — and what goes wrong when code stays trapped in notebook cells.

The Standard Python Project Layout

This layout is not a style preference — it's what pip, pytest, and every CI system on Earth expect by default. Adopt it and the whole ecosystem just works.

Project root
my_project/
← project root
├── pyproject.toml
project metadata + dependencies (PEP 621)
├── requirements.txt
optional lockfile — pyproject.toml alone is fine too
├── README.md
what this project does, how to install
├── Makefile
make train / make test / make lint
├── .gitignore
ignore data/, __pycache__, .venv, ...
├── docs/
documentation
├── src/
your Python source code
│ ├── data_prep/
preprocessing, dataset loading, transforms
│ │ ├── __init__.py
marks directory as a package
│ │ └── preprocessing.py
scale_features, encode_categorical, ...
│ ├── model/
model architectures
│ │ ├── __init__.py
public API for the model package
│ │ └── architecture.py
nn.Module subclasses, build_model(cfg)
│ ├── configs/
YAML hyperparameters
│ │ └── default.yaml
lr, batch_size, epochs, ...
│ ├── utils/
shared helpers: logging, seeding, path constants
│ │ ├── __init__.py
│ │ └── helpers.py
set_seed, get_logger, project_root, ...
│ └── pipeline.py
orchestration: load cfg → data → model → train
├── tests/
pytest test files (mirrors src/)
├── notebooks/
exploration — imports FROM src/, never the reverse
│ ├── 01_eda.ipynb
numbered prefix keeps ordering obvious
│ └── 02_prototype.ipynb
└── data/
datasets (gitignored — never commit)
🔍 About src/ and notebooks/

src/ vs root package. Many real projects (Django, Flask, Requests) skip src/ and put the package at the project root. Both are valid — the src/ version forces you to install before you can import, which surfaces install-time bugs earlier.

notebooks/ lives at the project root, not inside src/. Notebooks are exploration scratchpads; they should import from src/ (e.g. from model.architecture import build_model), never the reverse. Keeping them outside src/ keeps your installable package clean and prevents notebook-only junk from shipping to users.

The Notebook Trap

Notebooks are great for exploration. The moment you want to reuse, share, test, or deploy that code, six problems show up at once.

Can't import

Notebooks aren't modules. Reuse = copy-paste.

Can't run headless

No CLI, no python foo.py, no cron job.

Can't test

pytest can't reach cells. Notebook logic = untested.

Cells rot

Out-of-order runs poison state. Restart, things break.

Diffs are noise

JSON + images + exec counts. Code review impossible.

No configs

Hyperparameters baked into cells. Swap one = copy the file.

Notebook Code vs Project Code

The logic on both sides below is identical — load data, build a model, train it. The organisation is radically different. The right side is importable, configurable, testable, and version-controllable. The left side is none of those things.

Notebook — 20 scattered cells
train.ipynb
# Cell 1 — imports mixed with setup
import torch, yaml, os, numpy
from torch import nn

# Cell 3 — magic numbers buried in code
BATCH_SIZE = 64
LR = 0.001
EPOCHS = 10

# Cell 7 — data loading inline
train_ds = MyDataset(...)
train_dl = DataLoader(train_ds, ...)

# Cell 11 — model pasted from cell 9
class Net(nn.Module): ...
model = Net()

# Cell 14 — training loop
for epoch in range(EPOCHS):
    for x, y in train_dl: ...
Project — four clean files
clean run
# configs/train.yaml — hyperparameters live here
# src/data_prep/preprocessing.py  — get_dataloaders(cfg)
# src/model/architecture.py — build_model(cfg)
# src/pipeline.py — train(model, loaders, cfg)

import yaml
from data_prep.preprocessing  import get_dataloaders
from model.architecture import build_model
from pipeline import train

cfg = yaml.safe_load(open("configs/train.yaml"))
train_dl, val_dl = get_dataloaders(cfg)
model = build_model(cfg)
train(model, train_dl, val_dl, cfg)
Notebook

Scattered imports, hardcoded hyperparameters, model tangled with training, nothing importable. Change one experiment → edit cells and re-run by hand.

Project

Each concern in its own file. Hyperparameters in YAML. Three function calls orchestrate the whole pipeline. Change an experiment → swap a config file.

Section 2

Packaging mechanics

The machinery underneath: __init__.py, imports, pyproject.toml, YAML configs, and the pip install -e . command that ties everything together.

Module vs Package

These two words sound interchangeable but they're different things. Getting them straight makes every error message, every import, and every tutorial easier to read.

Module — a single .py file

Any file of Python code. The filename (without .py) is the module name.

on disk
my_utils.py
in Python
import my_utils
from my_utils import helper

Package — a directory with __init__.py

A folder marked with __init__.py. Can contain modules and sub-packages.

on disk
data_prep/
├── __init__.py
├── preprocessing.py
└── readers/
    ├── __init__.py
    └── csv.py
in Python
from data_prep import preprocessing
from data_prep.readers.csv import read_csv
💡 Rule of thumb

Ends in .py? It's a module. Directory with __init__.py inside? It's a package. In your project: pipeline.py is a module; src/data_prep/ is a package that contains the preprocessing module.

What Is a Python Package?

A package is any directory that contains a file named __init__.py. That's the whole rule. The file can be empty — its presence is what matters. Python uses it to decide "is this folder importable?"

with __init__.py
project layout
src/
└── my_project/
    ├── __init__.py   # even empty is fine
    └── data.py
# in Python:
from my_project.data import load   # ✓ works
without __init__.py
project layout
src/
└── my_project/
    └── data.py     # no __init__.py
# in Python:
from my_project.data import load
# ModuleNotFoundError:
# No module named 'my_project'

Every library you pip install — NumPy, PyTorch, pandas — is a package laid out this way, often with nested sub-packages (each with its own __init__.py).

__init__.py — Two Patterns

The file turns a directory into a package. What you put inside decides how users import from it. Two patterns — you'll use both in the same project.

Pattern 1 — Empty file (just the marker)

src/data_prep/__init__.py
# (empty — 0 bytes)
what users type
from data_prep.preprocessing
       import scale_features

Full path every time. Explicit. The safe default — start here.

Pattern 2 — Re-export the public API (with __all__)

src/model/__init__.py
# Re-export the blessed names:
from model.architecture import (
    Classifier,
    build_model,
)

# Public API — what users rely on.
__all__ = ["Classifier", "build_model"]
what users type
from model import Classifier

# the full path still works:
from model.architecture
       import Classifier
🎓 What __all__ means — in plain English

Think of __all__ as a sign on the package's front door: "these are the names we promise to keep."

  • Listed in __all__ → public. Safe to use; won't disappear in a minor version.
  • Not listed → internal. Might get renamed or deleted at any time — don't rely on it.
  • It also controls what from model import * pulls in (but import * is almost never used in real code).

Absolute vs Relative Imports

Two ways to refer to the same module. Absolute imports spell out the full path from the project root. Relative imports use dots. Both are legal; only one is recommended in most cases.

Absolute (preferred)

src/pipeline.py
from data_prep.preprocessing import get_dataloaders
from model.architecture import build_model
from utils.helpers import get_logger

Full path from the installed package root. Works from any file — scripts, notebooks, tests — no context needed. PEP 8 recommends absolute imports for public code.

Relative (same-package only)

src/data_prep/loaders.py
# single dot = same package (data_prep/)
from .preprocessing import get_dataloaders
from .dataset import MyDataset

# double dot = parent package (src/)
from ..utils.helpers import get_logger

Dots count directory levels: . = same package, .. = parent, ... = grandparent. Only resolves when the file is loaded as part of a package.

⚠️ Relative imports break if you run the file by path

Relative imports need the file to be loaded as part of a package. When you run Python with a file path, Python treats that file as a top-level script and __package__ is unset — so from .foo import bar raises ImportError: attempted relative import with no known parent package.

Note: this has nothing to do with pip install -e .. Editable install makes your package importable (via import data_prep), but running a file by path still bypasses the package system.

python src/data_prep/loaders.py ❌ file-path invocation — relative imports fail. python -m data_prep.loaders ✅ module invocation — data_prep is the parent package.

pyproject.toml — at a glance

The single configuration file for a modern Python project. Defined by PEP 621, read by every modern tool. It replaced the legacy trio of setup.py + setup.cfg + MANIFEST.in.

The file is organised into tables, each starting with a header in square brackets. The ones you'll meet most often:

pyproject.toml — the whole file at a glance
[build-system]                       # how pip builds your package
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"

[project]                             # metadata + runtime deps
name = "my_project"
version = "0.1.0"
dependencies = ["numpy", "torch", "pyyaml"]

[project.optional-dependencies]       # extras users opt into
dev = ["pytest", "pytest-cov", "ruff"]

[project.urls]                        # links shown on PyPI + `pip show`
Homepage = "https://github.com/you/my_project"

[project.scripts]                     # auto-creates a shell command on install
train = "pipeline:main"

[tool.setuptools.packages.find]       # LOOK inside src/ for packages to install
where = ["src"]                          # (src/ itself is NOT a package)

[tool.ruff]                            # ruff (linter + formatter) reads from here
line-length = 100

Each bracketed section is independent — you add, remove, or rearrange freely. The next few slides unpack each of these in turn.

[project] — metadata & runtime dependencies

The most-edited table in the file. It answers: what's this project called, what version is it, what's it about, who owns it, and what does it need to run? Only name and version are strictly required — everything else is optional but standard on real projects.

pyproject.toml — a fuller [project] section
[project]
name            = "my_project"
version         = "0.1.0"
description     = "A small, well-structured ML project."
readme          = "README.md"
requires-python = ">=3.10"
license         = {text = "MIT"}
keywords        = ["machine-learning", "pytorch", "training"]
authors         = [{name = "Your Name", email = "you@example.com"}]
maintainers     = [{name = "ML Team", email = "ml@example.com"}]

classifiers = [                              # PyPI's standard taxonomy
    "Development Status :: 3 - Alpha",
    "License :: OSI Approved :: MIT License",
    "Programming Language :: Python :: 3.10",
    "Programming Language :: Python :: 3.11",
    "Operating System :: OS Independent",
]

# Runtime dependencies — installed by `pip install .`
dependencies = [
    "numpy>=1.24",
    "torch>=2.0",
    "pyyaml>=6.0",
    "scikit-learn>=1.3",
]

[project.urls]                             # shown on PyPI + `pip show`
Homepage      = "https://github.com/you/my_project"
Repository    = "https://github.com/you/my_project"
Documentation = "https://my-project.readthedocs.io"
Issues        = "https://github.com/you/my_project/issues"

[project.scripts]                           # auto-creates shell commands
# After `pip install .`, typing `train` in ANY shell runs pipeline.main().
# Different from a Makefile target — Makefile aliases only work inside the
# repo and are for devs; these are real commands that ship with the package.
train    = "pipeline:main"
evaluate = "eval:main"
required

The only two keys you must set

  • name — the install-safe project name.
  • version — any PEP 440 string.
standard

Keys you'll always add

  • description, readme, authors — show up on PyPI and pip show.
  • requires-python — stops install on the wrong interpreter.
  • license — legal clarity; required for PyPI listings.
  • dependencies — what pip install . pulls in.
optional extras

Blocks that pay off

  • classifiers, keywords — discoverability on PyPI.
  • [project.urls] — links on the project page + pip show.
  • [project.scripts] — turns a function into a real CLI command after install.
  • maintainers — separate from authors once original author stops shipping.

[project.optional-dependencies] — extras

Dependencies your code doesn't need to run, but users might want in certain situations — testing tools, docs builders, GPU variants. Grouped into named extras that users opt into by name.

pyproject.toml
[project]
dependencies = ["torch", "numpy"]  # runtime

[project.optional-dependencies]
# group_name = [deps]
dev  = ["pytest", "pytest-cov", "ruff"]
docs = ["sphinx", "myst-parser"]
gpu  = ["torch[cuda]"]

# self-ref bundles every extra:
all  = ["my_project[dev,docs,gpu]"]
install with extras
# Runtime only:
$ pip install "."

# Runtime + dev tools:
$ pip install ".[dev]"

# Multiple groups at once:
$ pip install ".[dev,docs]"

# Every extra (via `all` group):
$ pip install ".[all]"

# From a PyPI-published package:
$ pip install "my_project[dev]"
💡 Common group names

dev (pytest + linters), docs (Sphinx), gpu (CUDA-flavoured deps), all (everything). The names are arbitrary — pick whatever matches your project's shape.

[build-system] & [tool.*] — the rest

Two more tables you'll see in almost every file. One tells pip how to turn your source into an installable package; the other is a shared home for every linter, formatter, and test runner's config.

build-system

How pip install . packages your source

[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"

# Alternatives:
#   "hatchling.build"   — modern, fast
#   "poetry.core.masonry.api"
#   "maturin"           — Rust extensions

What it does: when you run pip install ., pip doesn't natively know how to turn your source tree into a .whl. It reads requires, installs those into a fresh isolated build env, then calls the build-backend to do the actual packaging — gather files, write metadata, emit the wheel. Pick setuptools unless you have a reason not to; set it once, forget it.

tool.*

One file for every tool's config

# setuptools reads this during install:
[tool.setuptools.packages.find]
where = ["src"]

# pytest reads this when `pytest` runs:
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v --strict-markers"

# ruff reads this when `ruff check` runs:
[tool.ruff]
line-length = 100

# mypy reads this when `mypy src/` runs:
[tool.mypy]
strict = true

What it does: each tool looks for its own [tool.<its-name>] table and reads its config from there. Before pyproject.toml, every tool needed its own file — pytest.ini, .ruffrc, mypy.ini, setup.cfg, .coveragerc. Now they all share one file that travels with your project.

pip install -e . — The Editable Install

This is the one command that makes your own package importable from anywhere on your machine. Run it once from the project root; your code becomes import-able from any Python script or notebook, and edits take effect immediately.

pip install .

Regular install — copies your source

  1. Reads pyproject.toml + [build-system].
  2. Builds your package into a wheel (.whl).
  3. Copies the files into the env's site-packages/.
  4. Installs all declared dependencies.
Edit a source file? The copy in site-packages/ is now stale — you have to reinstall to see the change.
pip install -e .

Editable install — points to your source

  1. Reads pyproject.toml + [build-system].
  2. Registers the package metadata (no wheel copy of your code).
  3. Writes a pointer file in site-packages/ that redirects imports back to your source directory.
  4. Installs all declared dependencies.
Edit a source file? Python picks it up on the next import. No reinstall needed.
shell — run once, from the project root
$ cd my_project/

# Install the package in editable mode:
$ pip install -e "."

# Include the dev extras (pytest, ruff, etc.):
$ pip install -e ".[dev]"

# Verify — the location should be inside your project, NOT site-packages:
$ pip show my_project
Name: my_project
Version: 0.1.0
Location: /Users/you/projects/my_project/src
Editable project location: /Users/you/projects/my_project
💡 What ".[dev]" means

The [dev] part says "also install the optional extras named dev from pyproject.toml." Look at your [project.optional-dependencies] section — that's where dev = ["pytest", "pytest-cov", "ruff"] is defined. Square brackets = optional group.

Putting It All Together

The messy 100-line notebook is now a thin orchestration script that reads like a summary. All the logic lives in importable modules; the notebook just wires them together.

notebooks/run_training.ipynb
import yaml
from data_prep.preprocessing  import get_dataloaders
from model.architecture import build_model
from pipeline import train

# 1. Load config
cfg = yaml.safe_load(open("configs/train.yaml"))

# 2. Build data, model, and run training
train_dl, val_dl = get_dataloaders(cfg)
model   = build_model(cfg)
history = train(model, train_dl, val_dl, cfg)

# 3. Inspect
print("Final val accuracy:", history["val_acc"][-1])
Reusable

Any script, notebook, or test file can from model.architecture import ... and get the same functions. No copy-paste anywhere.

Configurable

Swap configs/train.yaml for configs/large.yaml and re-run — no code change. Git tracks every experiment as a diffable file.

Testable

Each function can be unit-tested in isolation with pytest (next class). Notebook-level testing was never possible; function-level testing is trivial.

🎓 The payoff

~100 lines of notebook became ~10 lines of orchestration. The remaining logic didn't disappear — it was promoted into modules that can now be reused, tested, versioned, and deployed. This exact pattern scales from a class assignment to a production codebase.

Makefile — Self-documenting Commands

A Makefile at the project root gives every common operation a short name. New teammates don't need to memorise flags; they type make train and it works. The file itself doubles as documentation.

Makefile
.PHONY: install train evaluate lint test clean

install:
	pip install -e ".[dev]"

train:
	python -m pipeline --config configs/train.yaml

evaluate:
	python -m pipeline --checkpoint outputs/best.pt

lint:
	ruff check src/ tests/

test:
	pytest tests/ -v

clean:
	rm -rf __pycache__ *.egg-info build/ dist/
	find . -type d -name __pycache__ -delete
shell
$ make install
$ make train
$ make test
why

Why Makefile?

Pre-installed on every Unix system. No extra tooling. The Makefile is short, version-controlled, and searchable — grep train Makefile tells you exactly what make train does.

windows

On Windows / alternatives

Use WSL, Git Bash, or swap the Makefile for a scripts/ folder with .ps1 / .sh files. Or try just — a modern cross-platform command runner with friendlier syntax.

YAML Configs — Separate What From How

YAML is a key–value format. When you load it in Python, you get back a regular dict. That's the whole idea — write your hyperparameters as text, load them as a dictionary, read them by key from your code.

configs/train.yaml
# key: value — like a dictionary
learning_rate: 0.001
batch_size:    64
epochs:        10

# Nesting works — inner dicts:
model:
  hidden:      128
  num_classes: 10
  dropout:     0.2
loading the config in Python
import yaml

with open("configs/train.yaml") as f:
    cfg = yaml.safe_load(f)

# cfg is just a Python dict:
print(type(cfg))
# <class 'dict'>

print(cfg["learning_rate"])   # 0.001
print(cfg["model"]["hidden"])  # 128
Why configs?

Every hyperparameter you might change lives in one file, separate from the code that uses it. Keep configs/small.yaml and configs/large.yaml side by side — same code, different experiments, just a CLI flag between them.

Safety

Always use yaml.safe_load. The plain yaml.load can execute arbitrary Python embedded in the file — same API, zero risk with safe_.

Summary & Going Deeper

Every project you build from now on starts with this structure — layout, pyproject.toml, editable install, YAML configs, Makefile. It's the baseline for everything that follows (testing, CI, model serving).
