Why Version Control?

Every project without version control eventually turns into chaos. Version control tracks every change, who made it, and when.

❌ Without version control

📄 report.py
📄 report_v2.py
📄 report_v2_final.py
📄 report_v2_final_FIXED.py
📄 report_USE_THIS_ONE.py
  • • Can't undo mistakes
  • • Who changed what?
  • • When did it break?

✅ With Git

📄 report.py — one file, full history
📋 47 commits · 3 branches
👥 3 contributors tracked
⏪ Revert to any point in time
  • • Every change tracked forever
  • • Safe collaboration
  • • Experiment freely

What is Git? What are GitHub, GitLab, Bitbucket?

These are not the same thing. Git is the tool. The others are places that host Git repositories.

Git

The version-control tool itself.

  • • A tool on your computer
  • • Tracks changes to files
  • • Works completely offline
  • • Free and open source

Hosting platforms

Websites that host Git repositories and add a UI on top.

GitHub
GitLab
Bitbucket
  • • Add code review, issues, CI/CD
  • • Need internet
  • • Same Git underneath — different UI & features

Same concept, different names. GitHub calls them Pull Requests; GitLab calls them Merge Requests. Both mean: "review my branch, then merge it."

"Git is the engine. GitHub, GitLab, and Bitbucket are different garages where you park and share."

Upstream tracking with -u — links your local branch to a remote branch so you don't have to type the full command every time.

Your local machine. The actual folder where your project files live. You edit files here normally. Git sees these changes as "untracked" or "modified" until you explicitly stage them.
Your local machine (inside .git/). A holding area where you select exactly which changes to include in the next commit. Think of it as a shopping cart — you add items before checkout.
Your local machine (the .git/ hidden folder). Contains every committed snapshot, the full project history, branches, and config. This folder IS your repository — delete it and all history is gone.
On the internet (GitHub, GitLab, etc.). A copy of your repository hosted online. This is how teams share code — everyone pushes to and pulls from the same remote to stay in sync.

Your First Repo & the Daily Cycle

Two ways to start a repo, then the four-command loop you'll run forever after.

Start fresh: git init

$ mkdir my-project && cd my-project
$ git init
Initialized empty Git repository

# Connect to GitHub:
$ git remote add origin URL
$ git push -u origin main

Existing: git clone

$ git clone https://github.com/
  user/repo.git
Cloning into 'repo'... done.

$ cd repo

# Remote "origin" auto-configured

Then every day: status → add → commit → push.

$ git status  # what changed?
modified: analysis.py   Untracked: utils.py

$ git add analysis.py utils.py  # stage
$ git commit -m "add utility functions for data cleaning"
$ git push origin main  # share
  • • git add . stages everything in the current dir — fast, but use with care
  • • Each commit is a snapshot — you can always go back to it
  • • -u on the first push sets the upstream so plain git push works after

Viewing History: log & diff

log = what happened. diff = what changed.

$ git log --oneline
a1b2c3d add utility functions for data cleaning
e4f5g6h fix: handle missing values in age column
i7j8k9l initial project setup
$ git diff
--- a/analysis.py
+++ b/analysis.py
- score = total / count
+ score = total / count if count > 0 else 0

Useful git log options

  • • git log --oneline — compact, one line per commit
  • • git log --graph — shows branch structure visually
  • • git log -5 — only the last 5 commits
  • • git log --author="name" — filter by author
  • • git log file.py — history for one file

Useful git diff options

  • • git diff — unstaged changes vs last commit
  • • git diff --staged — what you're about to commit
  • • git diff main..feature — between two branches
  • • git diff HEAD~3 — changes from 3 commits ago
  • • Green (+) = added. Red (−) = removed

Commit Messages

Your commit messages are how future-you and your teammates understand what happened and why.

❌ Bad

"fixed stuff"
"update"
"WIP"
"asdfgh"

✅ Good

"fix: handle missing values"
"add: CSV export for reports"
"refactor: simplify score logic"
"docs: update setup instructions"
  • • Start with a type: fix: add: update: remove: refactor: docs:
  • • Keep the first line under ~50 characters — this is the summary everyone reads
  • • Describe what changed and why, not how — the diff shows the code
  • • Each commit should be one logical change — don't bundle unrelated changes together

Branches: Concept & Commands

A branch is a parallel version of your project — work on a feature without touching stable code, then merge it back.

Each dot = one commit. Colored lines = branches. Where lines rejoin = a merge.

BranchPurpose
main Stable, production — permanent
feature-* New feature — delete after merge
fix-* Bug fix — delete after merge
  • • Branches are pointers to commits — creating one is instant
  • • One branch per task; never work directly on main
# See all branches
$ git branch
* main
feature-login

# Create + switch
$ git switch -c feature-cleanup

# Switch back
$ git switch main

# Delete merged branch
$ git branch -d feature-cleanup

Note: git switch is the modern command (Git 2.23+). Older tutorials use git checkout -b name / git checkout name — both work.

14 / 24

Merging

Merging brings changes from one branch into another. Switch to the target (usually main), then merge the feature in.

# Step 1: Go to the target branch
$ git switch main

# Step 2: Merge the feature in
$ git merge feature-cleanup
Merge made by the 'ort' strategy.
3 files changed, 22 insertions(+)

# Step 3: Clean up locally
$ git branch -d feature-cleanup

# Step 4: Delete the remote branch too
$ git push origin --delete feature-cleanup

A merge commit ties both histories together — it has two parents. Don't forget to push and delete the remote branch.

Getting Remote Changes: Fetch, Merge & Pull

A safe way (two steps) and a shortcut (one step).

Safe: fetch → review → merge

$ git fetch origin
# Downloaded, code untouched

$ git log origin/main --oneline
# See what's new

$ git merge origin/main
# Now apply it

Shortcut: pull

$ git pull origin main
# fetch + merge in one
git pull = git fetch + git merge

Convenient, but no chance to review before merging.

Team Workflow & Open Source

Everyone works on their own branch. Changes go through Pull Requests — reviewed before merging.

Team workflow (you have push access)

  1. Branch from main
  2. Commit your work on the branch
  3. Push the branch + open a PR
  4. Teammates review, you iterate
  5. Approved PR is merged into main

Open source (you don't own the repo)

  1. Fork on GitHub → github.com/you/repo
  2. Clone your fork
  3. Add upstream: git remote add upstream URL
  4. Branch, commit, push to your fork
  5. Open a PR from your fork to the original
Stay in sync: git fetch upstream → git merge upstream/main
Issues tracks bugs and feature requests right next to your code on GitHub / GitLab / Bitbucket. Reference them in commit messages and PR descriptions: "fix: handle null ages, fixes #12". When the PR merges, issue #12 auto-closes.

Git Best Practices

Rules that prevent 95% of Git disasters.

✅ Always Do

Commit often, keep commits small
Each commit = one logical change. Easier to review and revert.
One branch per feature/fix
Isolate your work. Never code directly on main.
Pull before you push
Get latest changes first to avoid conflicts.
Write meaningful commit messages
Your future self will thank you at 2 AM.
Review diff before committing
git diff --staged catches accidental changes.

❌ Never Do

Never commit secrets
API keys, passwords, .env — once pushed, in history forever.
Never commit large data files
CSVs, models, images — use .gitignore.
Never force push to main
--force overwrites everyone else's work.
Never rebase shared branches
Rewrites history others depend on.
Never git add . blindly
Always check git status first.

Merge Conflicts

When two branches change the same line, Git asks you to decide.

<<<<<<< HEAD (your branch)
score = total / count if count > 0 else 0
=======
score = total / max(count, 1)
>>>>>>> feature-branch (incoming)

How to resolve:

  1. Open the file and find the <<<< / ==== / >>>> markers
  2. Decide: keep yours, keep theirs, or write a combination
  3. Delete all the conflict markers — leave only the final code
  4. git add file.py — mark as resolved
  5. git commit — completes the merge (Git auto-generates a merge message)
  6. git push origin <branch> — share the resolved merge with the team

Recovering from Mistakes

Pick by where the change lives. Walk down the ladder: while it's still local, the reset family undoes things. Once it's pushed, only revert is safe.

Where the change isCommandWhat it does
Edited, not staged yetgit restore <file>Reverts file to last commit. ⚠️ Permanent
Staged, not committedgit restore --staged <file>Un-stages; edits stay in working dir
Committed, not pushed — keep the workgit reset --soft HEAD~1Un-commits; files re-staged
Committed, not pushed — drop the workgit reset --hard HEAD~1Un-commits AND erases edits
Already pushedgit revert <hash>New commit reverses the old. Never rewrites history.
Stop tracking a committed file (secrets, big data files)git rm --cached <file>Removes from git's index. File stays on disk.

Also useful: git stash / git stash pop — park work in progress and bring it back later (different from undo: nothing is being reverted, just temporarily set aside).

🚨 Pushed a secret

Rotate the key first, then untrack:

$ git rm --cached .env
$ echo ".env" >> .gitignore
$ git add .gitignore
$ git commit -m "remove secret"
$ git push origin main

⚠️ Wrong branch

Move the changes to a new branch:

$ git reset --soft HEAD~1
$ git stash
$ git switch -c correct-branch
$ git stash pop
$ git add <files>
$ git commit -m "msg"

💡 Pushed a bad commit

Don't rewrite history — revert:

$ git log --oneline
$ git revert a1b2c3d
$ git push origin main

Git Cheat Sheet

Bookmark this slide — every command you'll use, with what each one does.

📦 Setup

Configure git and start a repo

git init
Start a new repo in this folder
git clone <url>
Copy a remote repo to your machine
git remote add origin <url>
Link this repo to a remote (name it origin)
git config --global user.name "[name]"
Set the name stamped on every commit you make
git config --global user.email "[email]"
Set the email stamped on every commit

📋 Daily cycle

Stage, commit, share — every single day

git status
See what's changed, staged, and untracked right now
git add <file>
Stage this file for the next commit
git diff --staged
Preview exactly what the next commit will contain
git commit -m "[message]"
Save the staged changes with a short why-message
git push origin main
Send your local commits to the remote

🌳 Branching

Create, switch, merge, delete

git branch
List local branches; the current one is starred
git switch -c <name>
Create a new branch and switch to it in one step
git merge <branch>
Bring another branch into the current one
git branch -d <branch>
Delete a local branch (after it's been merged)
git push origin --delete <branch>
Delete the same branch on the remote too

☁️ Sync with remote

Bring remote changes in; send local out

git remote -v
Show the URLs this repo can fetch from and push to
git fetch origin
Download remote changes; don't apply them yet
git pull origin main
Fetch + merge into the current branch in one step
git pull --rebase origin main
Fetch + replay your commits on top of the new main
git push -u origin <branch>
First push of a new branch; sets upstream tracking

⏪ Undo (top → bottom = local → pushed)

Pick the right tool for where the bad change lives

git restore <file>
Discard unstaged edits to a file. ⚠️ Permanent
git restore --staged <file>
Un-stage a file but keep the edits in your working dir
git reset --soft HEAD~1
Un-commit the last commit; files stay staged
git reset --hard HEAD~1
Un-commit AND erase the edits. ⚠️ Permanent
git revert <hash>
New commit that undoes a pushed one — never rewrites history
git rm --cached <file>
Stop tracking a file; it stays on disk
git stash / git stash pop
Park work-in-progress, retrieve it later

🔍 Inspect

Read history; see what changed

git log --oneline
Compact commit history — one line per commit
git log --graph --all
Show all branches as an ASCII tree
git show <hash>
View the diff of a single commit
git blame <file>
See who last touched each line of a file
git branch -a
List all branches — both local and remote
