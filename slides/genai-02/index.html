RAG — Retrieval-Augmented Generation

Embeddings and vector databases

Why RAG exists

An LLM has no idea what's in your private documents. RAG fixes that by fetching relevant context at query time and pasting it into the prompt — without touching the model's weights.

Two motivations stack here: privacy (the model never trained on your docs) and freshness (your docs change daily; the model's training cut-off does not). Both are solved without retraining.

Full RAG pipeline

Indexing writes to the vector DB; query reads from it. Indexing runs once or on a schedule; query runs on every user request.

Embeddings

An embedding model is a separate model from your LLM. Its job is to map any piece of text to a vector of numbers in a space where semantically similar text lands near each other.

Cosine similarity

The score behind every dense retrieval. Cosine similarity measures the angle between two vectors — close to 1 when they point the same way, close to 0 when perpendicular, regardless of vector length.
SIMILARITY VS DISTANCE
cosine similarity ∈ [−1, +1]
cosine distance = 1 − cosine similarity ∈ [0, 2]
Vector DBs report distance; you convert back to similarity with 1 − distance.
Real text embeddings rarely go negative, so in practice you see similarities in roughly [0, 1].

Document parsing

Before chunking, every document has to become clean text. Garbage in → garbage retrieval — a sloppy parser will haunt every query you ever make.

.mdbuilt-in / markdown-itpreserve heading structure as metadata sections
.pdfpypdf · unstructuredmulti-column layouts · scanned PDFs (need OCR) · mangled tables

.docxpython-docxheadings and tables are structured — extract them as such
SANITY CHECK
Always print the first ~500 characters of every parsed document before you embed it. If it doesn't look readable to you, no LLM is going to make sense of it either. Mangled tables, lost headings, repeated boilerplate, and OCR typos all wreck embeddings silently.

Chunking strategies

A whole document is too big to embed and too big to paste into a prompt — the chunker splits it. There's no one right way to split: pick a strategy based on how your docs are structured and how users query them.
SENSIBLE DEFAULTS
Start with chunk_size=500, chunk_overlap=80. Too small loses context; too large blurs the embedding. Tune against a labelled eval set, not by eye.

Retrieval strategies

Dense — the default

Embedding similarity over an ANN index. Sublinear time, good recall on paraphrased queries.

Hybrid — add when keywords matter

Run BM25 in parallel and fuse the ranked lists. Rescues queries with exact identifiers, codes, or rare terms.

Rerank — add when quality > latency

Retrieve 20–100 cheap candidates, score each (query, chunk) pair with a cross-encoder, keep the top 3–10.

Query

Three logical inputs — the system rules, the retrieved chunks, and the user's question — get packed into the chat-completion message format: the system role carries the rules; the user role carries the chunks and the question, in that order.

Assembled prompt sent to the LLM

role: system
Answer ONLY from the context below.
Cite every fact as [source#chunk_id].
If the answer isn't there, say so plainly.

role: user
Context:
[refund_policy.html#refund_policy-c000]
Refunds within 30 days of purchase, item unused, original receipt presented.

[refund_policy.html#refund_policy-c001]
After 30 days, store credit may be offered at the manager's discretion.

Question: What is the refund window?
RETRIEVAL FINDS, THE LLM REASONS
Retrieval surfaces evidence; the LLM does the reasoning and synthesis over that evidence. The prompt above is what stitches the two halves into a single chat-completion call.

Citations and grounding

A grounded answer is one you can trace back to a source chunk. Without that trace, you can't audit the answer and your users can't trust it.
UNGROUNDED

"Our refund policy is 30 days."

Sounds confident. No way to tell whether the model read it in your docs or invented it. If the policy changed last week, you'd never know.

GROUNDED

"Our refund policy is 30 days (refund_policy.md, chunk 4)."

Same answer — auditable. A reviewer can open the source, find chunk 4, and verify in seconds. Wrong answer? Fix the doc, re-index, done.

HOW YOU GET CITATIONS
Tell the model in the system prompt to emit them, and hand each chunk in with its source + chunk_id already attached. The model can't cite what you didn't label.

Common RAG failure modes

A handful of failure modes show up over and over. Each one has a recognisable symptom and a specific fix — once you know the menu, debugging gets fast.
A

Wrong chunks retrieved

causechunk size too large, or query phrasing mismatches the docs. fixshrink chunks; embed query and a known-good chunk and inspect their similarity.
B

No relevant chunk retrieved

causethe doc never made it into the index, or chunks too small dropped key context. fixre-run ingestion; widen chunk size; verify the doc is in the DB by metadata filter.
C

Model ignores the context

causeweak system rule, or context buried after the question. fixtighten the system prompt; put context before the question; explicitly require citation.
D

Model hallucinates from context

causecontext is contradictory or sparse; model fills the gap with training data. fixrequire "I don't know" when context is insufficient; lower temperature.
E

Citation mismatch

causechunk ids in the prompt don't match what's stored, or model fabricates the citation. fixrender context with explicit [source::chunk_id] markers; validate citations against the retrieved set.
F

Context window overflow

causetop-k × chunk-size pushed past the LLM's window. fixcap top-k; consider smaller chunks; pick a model with a bigger context if it's genuinely needed.

Retrieval evaluation

Judge the retrieval before you judge the answer — if the right chunk never lands in the top-k, no prompt tuning saves you.
RECALL@K

Did we find them all?

Of the relevant chunks, what fraction made the top-k.

PRECISION@K

How much noise?

Of the k returned, what fraction was actually relevant.

HIT-RATE

Did we find it at all?

Did the correct chunk appear anywhere in the top-k — yes / no.

BUILD A SMALL LABELLED EVAL SET
20–50 questions paired with their expected chunks. Run as a CI check on every change to chunk_size, embedding model, or retrieval strategy — a hit-rate drop tells you which knob hurt, long before users complain.
